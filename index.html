Q1 FAT May 2024 (Premalatha M) 10 Marks 🔴
Illustrate the importance of the three-tier architecture for a University DBMS (managing student info, courses, faculty, academic records, administration).
▼
Question: You are hired as a database expert to design a comprehensive database system for a university. Illustrate the importance of the three-tier architecture for the given scenario.
Step 1 — Identify the Architecture
Since a university system has multiple user types (students, faculty, admins, managers), complex business logic (grade processing, scheduling), and a large backend database — the Three-Tier Architecture is the ideal choice. It separates Presentation, Logic, and Data into independent layers.
Step 2 — Three-Tier Architecture Diagram
┌─────────────────────────────────────────┐ │ TIER 1: PRESENTATION TIER │ │ (Client / User Interface Layer) │ │ │ │ [Web Browser] [Mobile App] [Desktop UI] │ │ Students Faculty Admins │ └──────────────┬──────────────────────────┘ │ HTTP / HTTPS Requests ▼ ┌─────────────────────────────────────────┐ │ TIER 2: APPLICATION TIER │ │ (Business Logic / Middle Layer) │ │ │ │ [Application Server] │ │ • Grade Calculation Logic │ │ • Course Enrollment Rules │ │ • Timetable Scheduling Engine │ │ • Authentication & Authorization │ │ • Fee Processing Logic │ └──────────────┬──────────────────────────┘ │ SQL / Database API Calls ▼ ┌─────────────────────────────────────────┐ │ TIER 3: DATA TIER │ │ (Database Server Layer) │ │ │ │ [DBMS — Oracle / MySQL / PostgreSQL] │ │ Tables: Students, Courses, Faculty, │ │ AcademicRecords, Admin │ └─────────────────────────────────────────┘
Step 3 — Explanation of Each Tier
Tier 1Presentation Tier: The front-end layer that students, faculty, and admins interact with. It collects input (e.g., enroll in course, view grades) and displays output. It contains NO business logic or direct DB access.
Tier 2Application Tier (Middle Tier): Contains all business rules. E.g., checking if a student has prerequisites before enrollment, calculating CGPA, validating faculty load. Acts as intermediary between UI and DB.
Tier 3Data Tier: The actual DBMS storing persistent data. Handles queries from Tier 2 only — never directly from Tier 1. Ensures data integrity, security, and concurrency.
Step 4 — Importance for University Scenario
Key Property Scalability + Security + Maintainability = 3-Tier Advantage
  • Scalability: Application servers can be replicated independently as student population grows.
  • Security: Database is never directly exposed to end users. Only the application server can access it.
  • Maintainability: Business logic can change (e.g., new grading policy) without touching the UI or DB schema.
  • Load Balancing: Multiple application servers can share load during exam season / registration rush.
  • Data Independence: UI changes don't affect DB; DB changes don't affect UI.
  • Concurrent Access: Thousands of students can simultaneously access marks, timetables, etc.
Conclusion: The three-tier architecture is the most appropriate for the university DBMS because it offers separation of concerns, improved security (DB not exposed), scalability for multiple users, and ease of maintenance when business rules change.
Q2 FAT Nov/Dec 2025 (Vijayakumar KP / Jani Anbarasi L) 10 Marks 🔴
Identify the DBMS component responsible for 5 given scenarios (1+1 marks each). Justify each.
▼
Scenarios: (i) Redefining table structures/updating catalog (ii) Generating inefficient query execution plans (iii) Managing bank account update failure and ensuring ACID (iv) Holding metadata (v) Executing compiled query plans and coordinating data retrieval.
DBMS Component Identification Strategy
1.Map each scenario to its DBMS subsystem using keyword clues.
2.State the component name clearly.
3.Justify in 1–2 sentences why that component handles it.
DBMS Components DDL Compiler | Query Optimizer | Recovery Manager | Data Dictionary | Query Processor
Answers (1 mark identify + 1 mark justify each)
# Scenario Component Justification
(i) Redefining table structures / updating catalog DDL Compiler The DDL (Data Definition Language) Compiler processes CREATE, ALTER, DROP commands. When a table structure is redefined, the DDL Compiler validates the definition and updates the System Catalog (Data Dictionary) with new metadata.
(ii) Generating inefficient query execution plans Query Optimizer The Query Optimizer analyzes the parsed query and generates execution plans by estimating cost (I/O, CPU). An inefficient plan means the optimizer chose a suboptimal join order or index strategy. The optimizer tries to find the least-cost plan, but may fail on complex queries.
(iii) Managing bank account update failure & ensuring ACID Transaction Manager / Recovery Manager The Transaction Manager ensures atomicity and consistency. If a transaction fails mid-way (e.g., server crash during debit), the Recovery Manager uses the log file to undo partial changes and restore the DB to a consistent state, ensuring ACID compliance.
(iv) Holding metadata (table attributes, user privileges, integrity constraints) Data Dictionary (System Catalog) The Data Dictionary stores metadata — information about the database itself: table schemas, attribute types, primary/foreign keys, user access rights, integrity constraints, and index information. It is queried by almost all other DBMS components.
(v) Executing compiled query plans and coordinating data retrieval Query Processor / Runtime Database Manager The Runtime Database Manager (part of the Query Processor) takes the optimized execution plan from the Query Optimizer and executes it — fetching pages from disk via the Buffer Manager, applying operators (join, select, project), and returning results to the user.

Q4 FAT Nov 2019 (Slot D1 & D2) 5 Marks 🟠
Detail all three architectures: Centralised, Two-Tier Client/Server, Three-Tier Client/Server DBMS.
▼
Architecture 1 — Centralised DBMS
All components — CPU, memory, DBMS software, and storage — reside on a single mainframe server. Users access it via dumb terminals (no processing at client side).
┌─────────────────────────────┐ │ CENTRALISED SERVER │ │ ┌───────┐ ┌──────────┐ │ │ │ DBMS │ │ Storage │ │ │ └───────┘ └──────────┘ │ └──────────┬──────────────────┘ │ ┌────────┴──────┐ │ Dumb │ │ Terminals │ (No local processing) └───────────────┘
  • All processing done at server
  • No data distribution — single point of failure
  • Simple to manage but not scalable
Architecture 2 — Two-Tier Client/Server
The system is split into two layers: Client (Tier 1) handles UI and some logic; Server (Tier 2) handles database processing.
┌───────────────────┐ ┌───────────────────┐ │ CLIENT (Tier 1) │ │ SERVER (Tier 2) │ │ - UI │◄────►│ - DBMS Engine │ │ - Application │ SQL │ - Database Files │ │ Logic │ │ - Query Processor│ └───────────────────┘ └───────────────────┘
  • "Fat client" — business logic at client side
  • Direct SQL calls from client to DB server
  • Problems: security (DB exposed), hard to update logic on all clients
Architecture 3 — Three-Tier Client/Server
[Client / Browser] ←HTTP→ [App Server] ←SQL→ [DB Server] Tier 1 (UI) Tier 2 (Logic) Tier 3 (Data)
  • "Thin client" — UI only at client
  • Business logic isolated in application server
  • Most secure, scalable, and maintainable
Comparison Centralised < 2-Tier < 3-Tier (in terms of scalability, security, flexibility)
Q5 FAT Apr/May 2025 (Sandhya) 5 Marks 🟠
Identify and describe the disadvantages of file processing system with suitable example [Context: Public distribution system for smart ration cards].
▼
Context
A public distribution system stores ration card data in separate flat files maintained by different government departments — ration, income tax, civil supplies.
Disadvantages of File Processing System
1.Data Redundancy & Inconsistency: Same citizen's data (name, address) stored in multiple files across departments. If address changes in one file, others remain outdated → inconsistency. E.g., ration department and income tax have different addresses for same person.
2.Difficulty in Data Access: No standard query language. A custom program must be written for every new query. E.g., "Find all BPL families in Chennai" requires writing a new program.
3.Data Isolation: Data scattered in different file formats (CSV, binary, text). Correlating data across files (e.g., linking ration card to income tax records) requires complex custom code.
4.Integrity Problems: No enforcement of constraints. E.g., a single family could be registered for two ration cards (both APL and BPL) — no system to prevent this.
5.Atomicity Failure: If power cuts mid-update (e.g., adding a new family member to ration card), file gets corrupted — no rollback mechanism.
6.Concurrent Access Issues: Multiple officers updating the same ration file simultaneously can cause data corruption — no locking mechanisms.
7.Security: No granular access control. Either the whole file is accessible or not — cannot restrict clerk to only their district's data.
Key Disadvantages Redundancy | Inconsistency | Isolation | No Atomicity | No Integrity | Poor Security

PerformanceInternal schema optimizes physical storage independently. Indexes, clustering, and page layouts can be tuned without changing the logical or view layers.

ER Diagram Design
General Algorithm — Drawing an ER Diagram
1.Identify all Entities (nouns in the scenario — things with independent existence).
2.Identify Attributes for each entity. Mark: key (underlined), composite, multivalued {}, derived ().
3.Identify all Relationships (verbs connecting entities). Name each relationship.
4.Determine Cardinality: 1:1, 1:N, M:N for each relationship.
5.Determine Participation Constraints: Total (double line) or Partial (single line).
6.Identify any Weak Entities (no key attribute — depend on owner entity).
7.Draw the diagram using standard ER notation: Rectangle=Entity, Ellipse=Attribute, Diamond=Relationship, Double Rectangle=Weak Entity.
ER Notation □ Entity | ◇ Relationship | ○ Attribute | ○ Derived attr | {○} Multivalued | ═══ Total Participation

MODULE 2
Extended ER (EER) — Generalisation, Specialisation, Aggregation
EER Concepts — Key Algorithm
1.Specialisation (Top-Down): Start with superclass → define subclasses with distinguishing features. E.g., Vehicle → {Car, Truck, Motorcycle}.
2.Generalisation (Bottom-Up): Start with multiple entity types → identify common attributes → form a higher-level superclass. E.g., {Car, Truck} → Vehicle.
3.Determine Disjoint (d) or Overlapping (o): Disjoint = entity belongs to only one subclass. Overlapping = entity can belong to multiple subclasses.
4.Determine Total or Partial: Total = every superclass entity must be in some subclass. Partial = not required.
IS-A Relationship Car IS-A Vehicle | Truck IS-A Vehicle | Subclass inherits ALL superclass attributes + adds own
Q1 FAT Nov 2019 (Slot D2) 2+2 Marks 🟡
Extend banking ER to include Savings Account (interest rate) and Checking Account (overdraft amount). Explain Generalisation and Specialisation.
▼
Extension — Adding Specialization to Account
[ACCOUNT] (account_id, balance, type) │ ══════════╪═══════════ ← Total Specialization │ {d} ← Disjoint ┌───────┴───────┐ │ │ [SAVINGS_ACCOUNT] [CHECKING_ACCOUNT] (interest_rate) (overdraft_amount)
Specialisation — Definition & Example
Specialisation is a top-down refinement process. We start with a higher-level entity type (Account) and define subgroups (Savings, Checking) based on distinguishing characteristics.
ProcessACCOUNT → {SAVINGS_ACCOUNT, CHECKING_ACCOUNT} based on account type attribute. Savings has interest_rate; Checking has overdraft_amount. Both inherit account_id and balance from ACCOUNT.
Specialisation reduces redundancy by storing common attributes only once in the superclass.
Generalisation — Definition & Example
Generalisation is a bottom-up process. We start with SAVINGS_ACCOUNT and CHECKING_ACCOUNT, identify common attributes (account_id, balance), and combine them into the higher-level superclass ACCOUNT.
Process{SAVINGS_ACCOUNT, CHECKING_ACCOUNT} → ACCOUNT (generalize). The common attributes are "lifted" to the superclass. This is essentially the reverse of specialisation.
Relational Mapping for EER (Specialisation) ACCOUNT(account_id PK, balance) | SAVINGS(account_id FK, interest_rate) | CHECKING(account_id FK, overdraft_amount)

ER-to-Relational Schema Mapping
ER → Relational Mapping Algorithm (7 Steps)
Step 1Regular Entity Types: Create a relation for each. PK = Key attribute of entity.
Step 2Weak Entity Types: Create a relation. PK = Owner's PK + Partial Key of weak entity.
Step 31:1 Relationships: Add FK of one relation into the other (prefer the total participation side). Or merge into one table.
Step 41:N Relationships: Add FK of the "1 side" entity into the "N side" relation.
Step 5M:N Relationships: Create a NEW relation with PKs of both entities as composite PK + any relationship attributes.
Step 6Multivalued Attributes: Create a NEW relation with entity PK + the multivalued attribute.
Step 7N-ary Relationships (n > 2): Create a new relation with PKs of all participating entities.
Key Mapping Rules 1:1 → merge or FK | 1:N → FK on N side | M:N → new table | Weak → owner_PK + partial_key

Normalization Algorithm — Step-by-Step
Step 11NF Check: No multivalued or composite attributes in any column. All values must be atomic. Remove repeating groups.

Functional Dependencies & Minimal Cover
Minimal Cover (Canonical Form) Algorithm
Step 1Right-Hand Side Simplification: Decompose all RHS into single attributes. E.g., A→BC becomes A→B and A→C.
Step 2LHS Redundancy Removal: For each FD with composite LHS, check if any LHS attribute is redundant. Remove if closure of remaining LHS still gives RHS.
Step 3Redundant FD Removal: For each FD X→A, check if A is in the closure of X using the remaining FDs (without this FD). If yes, the FD is redundant — remove it.
ResultThe minimal cover is the final reduced set of FDs.
Attribute Closure Algorithm X⁺ = X; Repeat: if ∃ Y→Z in F where Y⊆X⁺, then X⁺ = X⁺ ∪ Z; Until no change

Lossless Join (Non-Additive Join) Decomposition
Chase Algorithm — Lossless Join Test
Step 1Create a table (matrix) with rows = each decomposed relation, columns = attributes of original relation R.
Step 2Fill cells: if attribute is in the relation, put aᵢ (distinguished); otherwise put bᵢⱼ (non-distinguished).
Step 3Apply each FD: if two rows agree on LHS, make them agree on RHS by replacing bᵢⱼ values with aᵢ (prefer distinguished).
Step 4Repeat Step 3 until no more changes.
Step 5If any row becomes all aᵢ (all distinguished), the decomposition is lossless. Otherwise, it's lossy.
2-Relation Lossless Test R = R1 ∪ R2 is lossless iff: (R1∩R2 → R1−R2) OR (R1∩R2 → R2−R1)

MODULE 4
B+ Tree — Construction, Insertion, Deletion
B+ Tree Insertion Algorithm (Order m)
1.Insert key into appropriate leaf node (search from root down).
2.If leaf has ≤ m−1 keys: No split. Done.
3.If leaf has m keys (overflow): Split. Left node: ⌈m/2⌉ keys. Right node: remaining keys. Copy middle key up to parent (leaf node copies key up).
4.If parent also overflows: Split internal node. Left: ⌊m/2⌋ keys. Right: remaining. Push (not copy) middle key up to grandparent.
5.If root splits: create new root with the pushed-up key. Tree height increases by 1.
B+ Tree Deletion Algorithm
1.Find key in leaf. Delete it.
2.If leaf still has ≥ ⌈(m−1)/2⌉ keys: Done. Optionally update parent if deleted key was a separator.
3.If leaf has underflow (too few keys): Try to borrow from sibling. If sibling has extra keys, redistribute. Update separator in parent.
4.If cannot borrow: Merge with sibling. Pull down separator from parent. Parent may now underflow → recurse upward.
B+ Tree Order m Properties Max keys per node = m−1 | Min keys (non-root) = ⌈(m−1)/2⌉ | Max children = m | Min children (non-root internal) = ⌈m/2⌉

MODULE 4
Hashing — Static, Extendible, Linear Probing
Static Hashing Algorithm
1.Compute hash position: h(key) = key mod M (where M = table size).
2.If slot is empty: Insert at h(key).
3.If collision (slot occupied):
Linear Probing: Try h(key)+1, h(key)+2, ... (mod M) until empty slot.
Quadratic Probing: Try h(key)+1², h(key)+2², ... (mod M).
Extendible (Dynamic) Hashing Algorithm
1.Hash function: use last d bits of key's binary hash (global depth d).
2.Insert into bucket pointed to by d-bit prefix.
3.If bucket full: split. Increase local depth of bucket by 1. If local depth > global depth: double directory (increase global depth by 1).
4.Redistribute keys in split bucket using new bit.
Hash Functions Static: h(k) = k mod M | Linear Probing: h(k,i) = (h(k)+i) mod M | Quadratic: h(k,i) = (h(k)+i²) mod M

SQL Query Writing Strategy
1.Identify required columns (SELECT clause).
2.Identify required tables (FROM clause). Add JOINs if multiple tables.
3.Add filters (WHERE clause). Use AND/OR for multiple conditions.
4.Add grouping if counting/aggregating by a category (GROUP BY clause).
5.Add group filter if filtering aggregate results (HAVING clause).
6.Add ordering if needed (ORDER BY).
SQL Clause Order SELECT → FROM → JOIN → WHERE → GROUP BY → HAVING → ORDER BY → LIMIT

SQL on EMPLOYEES & DEPARTMENTS: (a) Salary 5000-12000 in dept 20/50 (b) No manager (c) Commission earners sorted (d) Highest/lowest/sum/avg salary per job (e) Employees hired before their managers.
▼
Schema Reference
EMPLOYEES (employee_id, last_name, first_name, email, hire_date, job_id, salary, commission_pct, manager_id, department_id)
DEPARTMENTS (department_id, department_name, manager_id, location_id)
(a)Display last name and salary of employees earning between 5000 and 12000 in dept 20 or 50. Label columns 'Employee' and 'Monthly Salary'.
SELECT last_name AS "Employee", salary AS "Monthly Salary" FROM employees WHERE salary BETWEEN 5000 AND 12000 AND department_id IN (20, 50) ORDER BY last_name;
(b)Report last name and job_id of employees with NO manager (manager_id is NULL).
SELECT last_name, job_id FROM employees WHERE manager_id IS NULL;
(c)Report last name, salary, commission of employees who earn commissions (commission_pct IS NOT NULL). Sort by salary descending.
SELECT last_name, salary, commission_pct FROM employees WHERE commission_pct IS NOT NULL ORDER BY salary DESC;
(d)Find highest, lowest, sum, average salary for each job_id (round to nearest whole).
SELECT job_id, MAX(salary) AS "Highest Salary", MIN(salary) AS "Lowest Salary", SUM(salary) AS "Total Salary", ROUND(AVG(salary)) AS "Average Salary" FROM employees GROUP BY job_id;
(e)Find employees hired before their managers. Display employee name and hire date.
SELECT e.last_name, e.hire_date FROM employees e, employees m WHERE e.manager_id = m.employee_id AND e.hire_date < m.hire_date;
