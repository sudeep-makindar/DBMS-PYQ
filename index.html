Q1 FAT May 2024 (Premalatha M) 10 Marks 🔴
Illustrate the importance of the three-tier architecture for a University DBMS (managing student info, courses, faculty, academic records, administration).
▼
Question: You are hired as a database expert to design a comprehensive database system for a university. Illustrate the importance of the three-tier architecture for the given scenario.
Step 1 — Identify the Architecture
Since a university system has multiple user types (students, faculty, admins, managers), complex business logic (grade processing, scheduling), and a large backend database — the Three-Tier Architecture is the ideal choice. It separates Presentation, Logic, and Data into independent layers.
Step 2 — Three-Tier Architecture Diagram
┌─────────────────────────────────────────┐ │ TIER 1: PRESENTATION TIER │ │ (Client / User Interface Layer) │ │ │ │ [Web Browser] [Mobile App] [Desktop UI] │ │ Students Faculty Admins │ └──────────────┬──────────────────────────┘ │ HTTP / HTTPS Requests ▼ ┌─────────────────────────────────────────┐ │ TIER 2: APPLICATION TIER │ │ (Business Logic / Middle Layer) │ │ │ │ [Application Server] │ │ • Grade Calculation Logic │ │ • Course Enrollment Rules │ │ • Timetable Scheduling Engine │ │ • Authentication & Authorization │ │ • Fee Processing Logic │ └──────────────┬──────────────────────────┘ │ SQL / Database API Calls ▼ ┌─────────────────────────────────────────┐ │ TIER 3: DATA TIER │ │ (Database Server Layer) │ │ │ │ [DBMS — Oracle / MySQL / PostgreSQL] │ │ Tables: Students, Courses, Faculty, │ │ AcademicRecords, Admin │ └─────────────────────────────────────────┘
Step 3 — Explanation of Each Tier
Tier 1Presentation Tier: The front-end layer that students, faculty, and admins interact with. It collects input (e.g., enroll in course, view grades) and displays output. It contains NO business logic or direct DB access.
Tier 2Application Tier (Middle Tier): Contains all business rules. E.g., checking if a student has prerequisites before enrollment, calculating CGPA, validating faculty load. Acts as intermediary between UI and DB.
Tier 3Data Tier: The actual DBMS storing persistent data. Handles queries from Tier 2 only — never directly from Tier 1. Ensures data integrity, security, and concurrency.
Step 4 — Importance for University Scenario
Key Property Scalability + Security + Maintainability = 3-Tier Advantage
  • Scalability: Application servers can be replicated independently as student population grows.
  • Security: Database is never directly exposed to end users. Only the application server can access it.
  • Maintainability: Business logic can change (e.g., new grading policy) without touching the UI or DB schema.
  • Load Balancing: Multiple application servers can share load during exam season / registration rush.
  • Data Independence: UI changes don't affect DB; DB changes don't affect UI.
  • Concurrent Access: Thousands of students can simultaneously access marks, timetables, etc.
Conclusion: The three-tier architecture is the most appropriate for the university DBMS because it offers separation of concerns, improved security (DB not exposed), scalability for multiple users, and ease of maintenance when business rules change.
Q2 FAT Nov/Dec 2025 (Vijayakumar KP / Jani Anbarasi L) 10 Marks 🔴
Identify the DBMS component responsible for 5 given scenarios (1+1 marks each). Justify each.
▼
Scenarios: (i) Redefining table structures/updating catalog (ii) Generating inefficient query execution plans (iii) Managing bank account update failure and ensuring ACID (iv) Holding metadata (v) Executing compiled query plans and coordinating data retrieval.
DBMS Component Identification Strategy
1.Map each scenario to its DBMS subsystem using keyword clues.
2.State the component name clearly.
3.Justify in 1–2 sentences why that component handles it.
DBMS Components DDL Compiler | Query Optimizer | Recovery Manager | Data Dictionary | Query Processor
Answers (1 mark identify + 1 mark justify each)
# Scenario Component Justification
(i) Redefining table structures / updating catalog DDL Compiler The DDL (Data Definition Language) Compiler processes CREATE, ALTER, DROP commands. When a table structure is redefined, the DDL Compiler validates the definition and updates the System Catalog (Data Dictionary) with new metadata.
(ii) Generating inefficient query execution plans Query Optimizer The Query Optimizer analyzes the parsed query and generates execution plans by estimating cost (I/O, CPU). An inefficient plan means the optimizer chose a suboptimal join order or index strategy. The optimizer tries to find the least-cost plan, but may fail on complex queries.
(iii) Managing bank account update failure & ensuring ACID Transaction Manager / Recovery Manager The Transaction Manager ensures atomicity and consistency. If a transaction fails mid-way (e.g., server crash during debit), the Recovery Manager uses the log file to undo partial changes and restore the DB to a consistent state, ensuring ACID compliance.
(iv) Holding metadata (table attributes, user privileges, integrity constraints) Data Dictionary (System Catalog) The Data Dictionary stores metadata — information about the database itself: table schemas, attribute types, primary/foreign keys, user access rights, integrity constraints, and index information. It is queried by almost all other DBMS components.
(v) Executing compiled query plans and coordinating data retrieval Query Processor / Runtime Database Manager The Runtime Database Manager (part of the Query Processor) takes the optimized execution plan from the Query Optimizer and executes it — fetching pages from disk via the Buffer Manager, applying operators (join, select, project), and returning results to the user.

Q4 FAT Nov 2019 (Slot D1 & D2) 5 Marks 🟠
Detail all three architectures: Centralised, Two-Tier Client/Server, Three-Tier Client/Server DBMS.
▼
Architecture 1 — Centralised DBMS
All components — CPU, memory, DBMS software, and storage — reside on a single mainframe server. Users access it via dumb terminals (no processing at client side).
┌─────────────────────────────┐ │ CENTRALISED SERVER │ │ ┌───────┐ ┌──────────┐ │ │ │ DBMS │ │ Storage │ │ │ └───────┘ └──────────┘ │ └──────────┬──────────────────┘ │ ┌────────┴──────┐ │ Dumb │ │ Terminals │ (No local processing) └───────────────┘
  • All processing done at server
  • No data distribution — single point of failure
  • Simple to manage but not scalable
Architecture 2 — Two-Tier Client/Server
The system is split into two layers: Client (Tier 1) handles UI and some logic; Server (Tier 2) handles database processing.
┌───────────────────┐ ┌───────────────────┐ │ CLIENT (Tier 1) │ │ SERVER (Tier 2) │ │ - UI │◄────►│ - DBMS Engine │ │ - Application │ SQL │ - Database Files │ │ Logic │ │ - Query Processor│ └───────────────────┘ └───────────────────┘
  • "Fat client" — business logic at client side
  • Direct SQL calls from client to DB server
  • Problems: security (DB exposed), hard to update logic on all clients
Architecture 3 — Three-Tier Client/Server
[Client / Browser] ←HTTP→ [App Server] ←SQL→ [DB Server] Tier 1 (UI) Tier 2 (Logic) Tier 3 (Data)
  • "Thin client" — UI only at client
  • Business logic isolated in application server
  • Most secure, scalable, and maintainable
Comparison Centralised < 2-Tier < 3-Tier (in terms of scalability, security, flexibility)
Q5 FAT Apr/May 2025 (Sandhya) 5 Marks 🟠
Identify and describe the disadvantages of file processing system with suitable example [Context: Public distribution system for smart ration cards].
▼
Context
A public distribution system stores ration card data in separate flat files maintained by different government departments — ration, income tax, civil supplies.
Disadvantages of File Processing System
1.Data Redundancy & Inconsistency: Same citizen's data (name, address) stored in multiple files across departments. If address changes in one file, others remain outdated → inconsistency. E.g., ration department and income tax have different addresses for same person.
2.Difficulty in Data Access: No standard query language. A custom program must be written for every new query. E.g., "Find all BPL families in Chennai" requires writing a new program.
3.Data Isolation: Data scattered in different file formats (CSV, binary, text). Correlating data across files (e.g., linking ration card to income tax records) requires complex custom code.
4.Integrity Problems: No enforcement of constraints. E.g., a single family could be registered for two ration cards (both APL and BPL) — no system to prevent this.
5.Atomicity Failure: If power cuts mid-update (e.g., adding a new family member to ration card), file gets corrupted — no rollback mechanism.
6.Concurrent Access Issues: Multiple officers updating the same ration file simultaneously can cause data corruption — no locking mechanisms.
7.Security: No granular access control. Either the whole file is accessible or not — cannot restrict clerk to only their district's data.
Key Disadvantages Redundancy | Inconsistency | Isolation | No Atomicity | No Integrity | Poor Security

Three-Schema Architecture
┌────────────────────────────────────────────┐ │ EXTERNAL SCHEMA LEVEL │ │ (Multiple Views for different user types) │ │ View 1: Student View (grades, timetable) │ │ View 2: Faculty View (courses, marks) │ │ View 3: Admin View (all data) │ └─────────────────┬──────────────────────────┘ │ Logical/External Mapping ┌─────────────────▼──────────────────────────┐ │ CONCEPTUAL SCHEMA LEVEL │ │ (Logical structure of entire database) │ │ Tables: Students, Courses, Faculty, │ │ Relationships, Constraints, Keys │ └─────────────────┬──────────────────────────┘ │ Internal/Conceptual Mapping ┌─────────────────▼──────────────────────────┐ │ INTERNAL SCHEMA LEVEL │ │ (Physical storage — how data is stored) │ │ B+ Trees, Hash files, page size, │ │ buffer size, index structures │ └────────────────────────────────────────────┘
How It Ensures:
ConsistencyConceptual schema enforces integrity constraints across all views — any data entered through any view must satisfy the same rules (e.g., student marks between 0–100).
SecurityExternal schema provides selective views — a student cannot access salary data of faculty. Access control is applied at the external level.
PerformanceInternal schema optimizes physical storage independently. Indexes, clustering, and page layouts can be tuned without changing the logical or view layers.
Data Independence Logical Independence: Change conceptual without changing external | Physical Independence: Change internal without changing conceptual

ER Diagram Design
General Algorithm — Drawing an ER Diagram
1.Identify all Entities (nouns in the scenario — things with independent existence).
2.Identify Attributes for each entity. Mark: key (underlined), composite, multivalued {}, derived ().
3.Identify all Relationships (verbs connecting entities). Name each relationship.
4.Determine Cardinality: 1:1, 1:N, M:N for each relationship.
5.Determine Participation Constraints: Total (double line) or Partial (single line).
6.Identify any Weak Entities (no key attribute — depend on owner entity).
7.Draw the diagram using standard ER notation: Rectangle=Entity, Ellipse=Attribute, Diamond=Relationship, Double Rectangle=Weak Entity.
ER Notation □ Entity | ◇ Relationship | ○ Attribute | ○ Derived attr | {○} Multivalued | ═══ Total Participation

Q5 FAT Nov/Dec 2025 (Vijayakumar KP / Jani Anbarasi L) – Q1 10 Marks 🔴
Identify which DBMS component is primarily responsible for: (i) Redefining table structures/updating catalog (ii) Generating inefficient query execution plan (iii) Coordinating multi-user access (iv) Translating high-level queries to low-level ops (v) Fetching data from disk to memory.
▼
#ScenarioComponentJustification
iRedefining table structures, updating catalogDDL CompilerProcesses DDL commands (CREATE/ALTER/DROP). Updates the System Catalog (Data Dictionary) with new schema info — table names, data types, constraints.
iiGenerating query execution planQuery OptimizerPart of Query Processor. Generates candidate execution plans and selects the best (based on indexes, statistics). A poor optimizer may choose full table scans over index scans — inefficient plans.
iiiCoordinating multi-user access, maintaining consistencyConcurrency Control ManagerUses 2PL, timestamp ordering, or MVCC to ensure isolation of concurrent transactions. Prevents lost updates, dirty reads, and ensures consistency (ACID-I and ACID-C).
ivTranslating high-level SQL to efficient low-level opsDML Compiler / Query ProcessorParses SQL → relational algebra → query plan. Works with optimizer for plan selection. Generates executable code (π, σ, ⋈ operations on physical files).
vFetching data from disk to memory, writing backBuffer ManagerManages buffer pool (RAM cache for disk pages). Fetches pages on demand (page faults). Uses LRU/MRU to evict pages. Writes dirty pages to disk. Critical for I/O performance.

MODULE 2
Extended ER (EER) — Generalisation, Specialisation, Aggregation
EER Concepts — Key Algorithm
1.Specialisation (Top-Down): Start with superclass → define subclasses with distinguishing features. E.g., Vehicle → {Car, Truck, Motorcycle}.
2.Generalisation (Bottom-Up): Start with multiple entity types → identify common attributes → form a higher-level superclass. E.g., {Car, Truck} → Vehicle.
3.Determine Disjoint (d) or Overlapping (o): Disjoint = entity belongs to only one subclass. Overlapping = entity can belong to multiple subclasses.
4.Determine Total or Partial: Total = every superclass entity must be in some subclass. Partial = not required.
IS-A Relationship Car IS-A Vehicle | Truck IS-A Vehicle | Subclass inherits ALL superclass attributes + adds own

Normalization (1NF → 2NF → 3NF → BCNF)
Normalization Algorithm — Step-by-Step
Step 11NF Check: No multivalued or composite attributes in any column. All values must be atomic. Remove repeating groups.
Step 2Find Candidate Key: Use attribute closure algorithm to find all candidate keys.
Step 32NF Check: No partial dependency (non-prime attribute → part of candidate key). Only applies if CK is composite. Remove partial dependencies into separate tables.
Step 43NF Check: No transitive dependency (non-prime → non-prime). A→B and B→C where B is not a key. Remove transitive FDs into separate tables.
Step 5BCNF Check: For every FD X→Y, X must be a superkey. Stronger than 3NF. Decompose if violated.
Normal Forms 1NF: No multi-valued | 2NF: 1NF + No Partial Dep | 3NF: 2NF + No Transitive Dep | BCNF: Every Determinant is Superkey

Functional Dependencies & Minimal Cover
Minimal Cover (Canonical Form) Algorithm
Step 1Right-Hand Side Simplification: Decompose all RHS into single attributes. E.g., A→BC becomes A→B and A→C.
Step 2LHS Redundancy Removal: For each FD with composite LHS, check if any LHS attribute is redundant. Remove if closure of remaining LHS still gives RHS.
Step 3Redundant FD Removal: For each FD X→A, check if A is in the closure of X using the remaining FDs (without this FD). If yes, the FD is redundant — remove it.
ResultThe minimal cover is the final reduced set of FDs.
Attribute Closure Algorithm X⁺ = X; Repeat: if ∃ Y→Z in F where Y⊆X⁺, then X⁺ = X⁺ ∪ Z; Until no change

Lossless Join (Non-Additive Join) Decomposition
Chase Algorithm — Lossless Join Test
Step 1Create a table (matrix) with rows = each decomposed relation, columns = attributes of original relation R.
Step 2Fill cells: if attribute is in the relation, put aᵢ (distinguished); otherwise put bᵢⱼ (non-distinguished).
Step 3Apply each FD: if two rows agree on LHS, make them agree on RHS by replacing bᵢⱼ values with aᵢ (prefer distinguished).
Step 4Repeat Step 3 until no more changes.
Step 5If any row becomes all aᵢ (all distinguished), the decomposition is lossless. Otherwise, it's lossy.
2-Relation Lossless Test R = R1 ∪ R2 is lossless iff: (R1∩R2 → R1−R2) OR (R1∩R2 → R2−R1)

MODULE 4
B+ Tree — Construction, Insertion, Deletion
B+ Tree Insertion Algorithm (Order m)
1.Insert key into appropriate leaf node (search from root down).
2.If leaf has ≤ m−1 keys: No split. Done.
3.If leaf has m keys (overflow): Split. Left node: ⌈m/2⌉ keys. Right node: remaining keys. Copy middle key up to parent (leaf node copies key up).
4.If parent also overflows: Split internal node. Left: ⌊m/2⌋ keys. Right: remaining. Push (not copy) middle key up to grandparent.
5.If root splits: create new root with the pushed-up key. Tree height increases by 1.
B+ Tree Deletion Algorithm
1.Find key in leaf. Delete it.
2.If leaf still has ≥ ⌈(m−1)/2⌉ keys: Done. Optionally update parent if deleted key was a separator.
3.If leaf has underflow (too few keys): Try to borrow from sibling. If sibling has extra keys, redistribute. Update separator in parent.
4.If cannot borrow: Merge with sibling. Pull down separator from parent. Parent may now underflow → recurse upward.
B+ Tree Order m Properties Max keys per node = m−1 | Min keys (non-root) = ⌈(m−1)/2⌉ | Max children = m | Min children (non-root internal) = ⌈m/2⌉

Hashing — Static, Extendible, Linear Probing
Static Hashing Algorithm
1.Compute hash position: h(key) = key mod M (where M = table size).
2.If slot is empty: Insert at h(key).
3.If collision (slot occupied):
Linear Probing: Try h(key)+1, h(key)+2, ... (mod M) until empty slot.
Quadratic Probing: Try h(key)+1², h(key)+2², ... (mod M).
Extendible (Dynamic) Hashing Algorithm
1.Hash function: use last d bits of key's binary hash (global depth d).
2.Insert into bucket pointed to by d-bit prefix.
3.If bucket full: split. Increase local depth of bucket by 1. If local depth > global depth: double directory (increase global depth by 1).
4.Redistribute keys in split bucket using new bit.
Hash Functions Static: h(k) = k mod M | Linear Probing: h(k,i) = (h(k)+i) mod M | Quadratic: h(k,i) = (h(k)+i²) mod M

Question: What do you think are the disadvantages of multi-level indexing? Detail how dynamic multi-level indexes address the issues of multi-level indexes with suitable examples.
Algorithm — Answering Multi-level Indexing Questions
1.Define multi-level indexing and why it was introduced (problem it solves over single-level).
2.List the disadvantages of static multi-level indexing clearly.
3.Explain how dynamic multi-level indexes (B+ Trees) overcome each disadvantage.
4.Give a concrete example with a small B+ Tree diagram.
5.Conclude with key advantages of the dynamic approach.
Key Formula Levels in Multi-level Index = ⌈logfo(n)⌉  |  fo = fan-out (index blocking factor)  |  n = number of records
Step 1 — What is Multi-Level Indexing?
A multi-level index is created when a single-level index itself becomes too large to fit in memory. The idea is to build an index over the index file — Level 1 (primary index over data), Level 2 (index over Level 1), and so on, until the top-level fits in a single disk block.
Data File → Level 1 Index → Level 2 Index → Level 3 Index (all records) (sparse, 1/block) (sparse, 1/block) (fits in RAM) n records n/fo entries n/fo² entries n/fo³ entries
Example: n=1,000,000 records, fo=100  →  L1: 10,000 entries | L2: 100 entries | L3: 1 entry (1 block in RAM). Only 3+1=4 disk reads instead of ⌈log₂ 10,000⌉=14!
Step 2 — Disadvantages of Static Multi-Level Indexing
1.Overflow and Periodic Reorganization: When new records are inserted, index blocks may become full, creating overflow blocks. To fix this, the entire data file and all index levels must be reorganized periodically — an extremely expensive operation that can take hours on large databases.
2.Rigid Structure — Fixed Levels: The number of levels is fixed at creation time. As the database grows beyond the original design, performance degrades because new levels cannot be added dynamically without complete reconstruction.
3.Wasted Space After Deletions: After records are deleted, their index entries may remain (orphan/dangling entries). Index blocks become partially empty (low occupancy) and space cannot be reclaimed without full reorganization.
4.Insertion Inefficiency: Inserting a new record in the sorted data file requires physically shifting records to maintain sorted order, which invalidates many index pointers across all index levels — requiring expensive updates.
5.Unbalanced Performance: After many insertions and deletions, the structure becomes unbalanced — some index paths may traverse more levels than others, causing unpredictable and degrading search performance.
Problems Summary Static Multi-Level: Overflow | Reorganization | Rigid Levels | Space Waste | Slow Insert/Delete | Unbalanced
Step 3 — Dynamic Multi-Level Index (B+ Tree) Solves All Issues
A B+ Tree (Balanced Plus Tree) is the standard dynamic multi-level index. It automatically adjusts its structure with every insertion and deletion, always maintaining balance. All data pointers live in leaf nodes; internal nodes serve as a navigational index.
Solves #1No Reorganization: When a node is full during insertion, it splits — divides into two nodes and promotes the middle key upward. No overflow blocks and no global reorganization is ever needed.
Solves #2Self-Balancing Dynamic Levels: B+ Trees are always height-balanced. All leaf nodes are at the exact same depth. The tree grows upward (a new root is created when the root splits) and shrinks downward (root is removed when it has only one child). This is automatic and transparent.
Solves #3Enforced Minimum Occupancy: Every non-root node must be at least 50% full (minimum ⌈p/2⌉ pointers where p = order). After deletion, if a node falls below 50%, it either borrows a key from a sibling (redistribution) or merges with a sibling, reclaiming space automatically.
Solves #4Efficient Insertion O(log n): Data file does not need to be physically sorted. B+ Trees only require the tree nodes to maintain sorted order internally. Insertions cost O(log n) at most — no shifting of data records.
Solves #5Guaranteed O(log n) Search: Since the tree is always balanced, every search follows exactly h = height steps from root to leaf, where h = O(logp n). Performance is always predictable and optimal.
Step 4 — Example: B+ Tree of Order 3 (Dynamic Multi-Level)
Order p=3: each internal node → max 2 keys, 3 pointers. Each leaf → max 2 key-pointer pairs + 1 pointer to next leaf. Insert keys: 10, 20, 30, 40, 50.
Insert 10, 20: Leaf = [10, 20] Insert 30 → Leaf [10,20,30] is FULL (max=2 for order 3): SPLIT → Promote 20 to new root: [20] / \ [10, 20] → [30] ← leaves linked with → Insert 40: Goes to right leaf → [30, 40] [20] / \ [10, 20] → [30, 40] Insert 50 → [30,40,50] FULL → Split, promote 40: [20 | 40] / | \ [10,20] → [30,40] → [50] ← all leaves linked ✓ Always balanced: all leaves at same level ✓ Range query: traverse linked leaf list (→) ✓ Point query: root → internal node → leaf — always 2 hops (height=2)
B+ Tree Properties Order p: max p pointers per node | min ⌈p/2⌉ pointers (non-root) | All data at leaf level | Leaves linked for range queries | Height h = ⌈log⌈p/2⌉(n)⌉
Step 5 — Comparison: Static vs Dynamic Multi-Level Index

DeletionOrphan entries; periodic cleanup neededRedistribution / merge — O(log n)

Conclusion: Static multi-level indexes suffer from reorganization overhead, rigidity, wasted space, and degrading performance. Dynamic multi-level indexes (B+ Trees) solve all these problems through automatic node splitting/merging and self-balancing, guaranteeing O(log n) performance for all operations. This is why B+ Trees are the universal choice for database indexing in all major DBMS.

Algorithm — Constructing a Multi-Level Index (Step-by-Step)
1.Identify the data file blocks and their anchor keys (smallest key in each block).
2.Build Level 1 index — one entry (anchor key, pointer) per data block. Group into index blocks using fo (fan-out = keys per index block).
3.Build Level 2 index — one entry per Level 1 block. Group using same fo.
4.Build Level 3 index — one entry per Level 2 block. Stop when top-level fits in 1 block.

Relational Algebra Transformation Rules
These rules are used in heuristic query optimization to transform a query into a more efficient equivalent form. They are equivalence-preserving transformations.
Rule 1Cascade of Selections (σ splitting): σc1 ∧ c2(R) ≡ σc1(σc2(R)). A conjunction of conditions can be split into a cascade of individual selections. Use: Push selections as far down the tree as possible.
Rule 2Commutativity of Selection: σc1(σc2(R)) ≡ σc2(σc1(R)). The order of selections can be swapped freely.
Rule 3Cascade of Projections: πL1(πL2(...(πLn(R))...)) ≡ πL1(R), where L1 ⊆ L2 ⊆ ... ⊆ Ln. Only the final (outermost) projection matters.
Rule 4Commuting Selection with Projection: σc(πL(R)) ≡ πL(σc(R)), when condition c involves only attributes in L. Apply selection before projection to reduce tuples first.
Rule 5Commutativity of Join/Cartesian Product: R ⋈c S ≡ S ⋈c R. Join order can be swapped. Use: Place smaller/more-filtered relation on the left for efficiency.
Rule 6Associativity of Join: (R ⋈ S) ⋈ T ≡ R ⋈ (S ⋈ T). Joins can be re-grouped. Use: Choose join order that produces smallest intermediate results.
Rule 7Commuting Selection with Cartesian Product: σc(R × S) ≡ R ⋈c S. A selection on a cross product converts it to a join — much more efficient.
Rule 8Commuting Selection with Set Operations: σc(R ∪ S) ≡ σc(R) ∪ σc(S). Push selection into both operands of union/intersection/difference.
Rule 9Commuting Projection with Join: πL1∪L2(R ⋈c S) ≡ (πL1(R)) ⋈c (πL2(S)), where c uses only attributes in L1 and L2. Reduce columns before joining.
Key Heuristics (exam tip) 1. Push σ down (apply filters early) | 2. Push π down (reduce columns early) | 3. Replace σ(R×S) with ⋈ | 4. Choose most restrictive join first
Summary: The 9 transformation rules allow a DBMS query optimizer to rearrange relational algebra expressions without changing their result, but in a more efficient order — primarily by reducing the size of intermediate relations through early selection and projection.

MODULE 5
Log-based Recovery (Deferred & Immediate Update)
Algorithm — Writing Log Records for Recovery
1.List all operations in schedule order.
2.Deferred Update: Defer all writes to disk until commit. Log format: <Ti, start> → <Ti, X, new_value> → <Ti, commit>. On recovery: REDO committed Tx only. Never UNDO (never written to DB before commit).
3.Immediate Update: Write to DB immediately (before commit). Log format: <Ti, start> → <Ti, X, old_value, new_value> → <Ti, commit>. On recovery: REDO committed Tx, UNDO uncommitted Tx.
4.On crash: Scan log. Committed Tx → REDO (re-apply writes). Uncommitted Tx → UNDO (roll back using old_value).
Log Record Formats Deferred: <Ti, start>, <Ti, X, new_val>, <Ti, commit> | Immediate: <Ti, start>, <Ti, X, old_val, new_val>, <Ti, commit>

Step 1 — Atomicity (All or Nothing)
ProblemAfter crash: Current was debited (₹5000 deducted) but Savings was NOT credited. DB is in inconsistent state — money "disappeared".
SolutionUse Log-based recovery with UNDO. The log contains <T1, Current, old_value, new_value> and <T1, Savings, old_value, new_value> (for immediate) or <T1, Savings, new_value> (deferred). RuleIntegrity constraint: Current_Balance + Savings_Balance must remain constant (total money conserved). Before: C + S = X. After transfer attempt: C reduced but S not increased → Total ≠ X → Inconsistent!
SolutionAtomicity ensures either BOTH writes happen (consistent) or NEITHER happens (consistent). The DBMS checks integrity constraints after commit — but since T1 never commits, the constraint violation is avoided by UNDO.

ProblemIf another transaction T2 reads Current_Balance after the debit but before crash recovery, it sees the incorrect reduced balance (dirty read).

Step 4 — Durability
GuaranteeOnce T1 successfully commits, the transfer (both debit and credit) must survive any subsequent crash. Cascadeless & Recoverable Schedules
Algorithm — Checking Recoverable / Cascadeless Schedules
1.List all transactions and their operations in schedule order.
2.Identify all dirty reads: Tj reads X written by Ti before Ti commits → Ti must commit before Tj commits.
3.Recoverable: ∀ dirty reads Tj reads from Ti → Ti commits before Tj. If not, add commit ordering.

MODULE 6
Two-Phase Locking (2PL) Protocol
Algorithm — 2PL Protocol
1.Growing Phase: Transaction acquires locks (shared S for Read, exclusive X for Write). Cannot release any lock in this phase.
2.Shrinking Phase: Transaction releases locks. Cannot acquire any new lock in this phase.
3.Lock Point: The moment the last lock is acquired (end of growing phase). Used for ordering serializable schedule.
4.Basic 2PL: Locks released after lock point (can still be before commit). May cause cascading aborts.
5.Strict 2PL: Exclusive (X) locks held until commit. Prevents dirty reads.
6.Rigorous 2PL: ALL locks (S and X) held until commit. Prevents all anomalies. Serial order = commit order.
Lock Compatibility Matrix S + S = Compatible ✓ | S + X = Not Compatible ✗ | X + S = Not Compatible ✗ | X + X = Not Compatible ✗

1.Each transaction Ti has timestamp TS(Ti) = time it starts. Older transaction = smaller timestamp.
2.Each data item X has: Read_TS(X) = largest TS of Tx that read X. Write_TS(X) = largest TS of Tx that wrote X.
3.Read(X) by Ti: If TS(Ti) < Write_TS(X) → REJECT (Ti is too old, X was already written by newer Tx) → rollback Ti. Else: Execute read, Read_TS(X) = max(TS(Ti), Read_TS(X)).
4.Write(X) by Ti: If TS(Ti) < Read_TS(X) → REJECT (newer Tx already read X). If TS(Ti) < Write_TS(X) → REJECT (newer Tx already wrote X). Else: Execute write, Write_TS(X) = TS(Ti).
5.Rejected transaction is rolled back and restarted with a new (higher) timestamp.
Timestamp Rules Read(X) rejected if TS(Ti) < Write_TS(X) | Write(X) rejected if TS(Ti) < Read_TS(X) OR TS(Ti) < Write_TS(X)

ACID Properties of Transactions
AtomicityA transaction is an indivisible unit — either ALL its operations complete successfully (commit) or NONE do (rollback). No partial execution. Example: Bank transfer either debits AND credits, or does neither.
ConsistencyA transaction must transform the database from one consistent state to another. All integrity constraints (primary key, foreign key, domain) must hold before and after the transaction.
IsolationConcurrent transactions execute as if they were serial (one after another). Intermediate states of a transaction are invisible to other concurrent transactions.
DurabilityOnce a transaction commits, its changes are permanent — even in the event of system failures. Achieved through logging and checkpointing.
Need for Locks in Concurrent Execution
Without locks, concurrent transactions may produce incorrect results:
Lost UpdateT1 reads X=100, T2 reads X=100. T1 writes X=150. T2 writes X=120. T1's update is lost! Correct result should have been X=170 (100-50+70 for example).
Dirty ReadT1 writes X=200, T2 reads X=200, T1 aborts → X reverts to 100. T2 has read a value that no longer exists — inconsistent computation.
SolutionLocks ensure that when T1 is modifying X, no other transaction can read or write X until T1 finishes. This enforces isolation.

1. Lost Update Problem
Definition: Two transactions read the same item, compute new values based on the old value, and both write back. The second write overwrites the first, making the first transaction's update permanently lost.
Time T1 (books seat 5A) T2 (books seat 5A) 1 Read(Seat5A) = EMPTY 2 Read(Seat5A) = EMPTY 3 Write(Seat5A = BOOKED) 4 Write(Seat5A = BOOKED) 5 Commit 6 Commit Result: Both T1 and T2 think they booked 5A. T1's booking appears committed but T2's write OVERWROTE it. → Double booking! T1's update is LOST.
Lost Update Pattern T1 R(X) → T2 R(X) → T1 W(X) → T2 W(X) → T1's write is overwritten (lost) by T2
Solution: Exclusive lock on the data item during the entire read-modify-write cycle.
2. Temporary Update (Dirty Read) Problem
Definition: T1 updates a data item but then aborts (rolls back). Before the rollback, T2 reads the updated (dirty) value. T2 now has incorrect data.
Time T1 (transfer Rs.500) T2 (check balance) 1 Read(A) = 1000 2 A := A - 500 3 Write(A) = 500 4 Read(A) = 500 ← DIRTY READ! 5 [System Crash → T1 ABORTS] 6 A restored to 1000 7 Uses A = 500 (WRONG!) Commits with incorrect data.
Dirty Read Pattern T1 W(X) → T2 R(X) → T1 ABORT (rollback) → T2 committed with wrong X value
Solution: Strict 2PL — T2 can only read X after T1 commits (X lock held by T1 until commit).
3. Incorrect Summary Problem
Definition: T1 is computing an aggregate (sum/average) over multiple data items while T2 is concurrently updating some of those items. T1 reads some items before T2's update and some after — producing an incorrect aggregate.
Accounts: A=100, B=200, C=300. Total should = 600. T1: compute SUM(A+B+C) T2: transfer 50 from B to C Time T1 (sum) T2 (transfer) 1 read(A) = 100 2 Read(B)=200; B:=200-50; Write(B)=150 3 Read(C)=300; C:=300+50; Write(C)=350 4 Read(B) = 150 ← (after T2 updated) 5 Read(C) = 300 ← (before T2's C update flushed?) 6 SUM = 100+150+300 = 550 ← WRONG! Should be 600
Incorrect Summary Pattern T1 reads items at different times while T2 is updating them → inconsistent snapshot → wrong aggregate result
Solution: Snapshot isolation or shared locks on ALL items being read before T1 starts computing the aggregate.
All Three Problems Summary: (1) Lost Update — T2 overwrites T1's committed write. (2) Dirty Read — T2 reads T1's uncommitted (dirty) write. (3) Incorrect Summary — T1 sees partial effects of T2's concurrent update. All are solved by proper locking protocols (2PL) or isolation levels (Serializable).

CAP Theorem and NoSQL Relevance
CAP Theorem (proposed by Eric Brewer, 2000) states that a distributed data store can guarantee at most 2 of 3 properties. Traditional RDBMS (SQL) prioritizes CA (ignoring P, run on single node). NoSQL databases were designed for distributed environments — they must handle P, so they choose between C and A.

Document StoreMongoDBCPStrong consistency by default; may be unavailable during partition

Why P is Almost Always Required in NoSQL
NoSQL databases are almost always distributed across multiple nodes/data centers. Network partitions (failures) are inevitable in such systems. Therefore, NoSQL always includes Partition Tolerance (P) — making the real choice between C (CP systems) and A (AP systems).
Eventual Consistency (AP choice): Many NoSQL systems use BASE (Basically Available, Soft-state, Eventually consistent) instead of ACID. Data replication happens asynchronously — all nodes converge to the same state eventually, but at any instant, different nodes might show different values.
NoSQL + CAP All NoSQL systems must choose: CP (consistency + partition) = strong consistency, possible unavailability | AP (availability + partition) = always responds, possible stale data
Relevance: CAP theorem is the foundational design principle for all NoSQL databases. It explains why Cassandra (Instagram, Netflix) chooses AP for high-traffic read/write workloads, while HBase (Facebook Messages) chooses CP for message ordering integrity. Developers must understand CAP to choose the right NoSQL database for their application's requirements.
